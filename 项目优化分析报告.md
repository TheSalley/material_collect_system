# 素材收集系统 — 项目优化分析报告

> 生成日期：2026-年5月19日  
> 项目规模：82 个 Vue 组件 + 21 个 JS 文件，总计约 17,054 行代码  
> 技术栈：Vue 3.5 + Vite 7 + Element Plus 2.11 + Tailwind CSS 4 + Pinia 3 + Vue Router 4

---

## 一、严重问题（建议优先修复）

### 1.1 Tailwind CSS v4 配置文件不兼容

**问题**：`taildwindcss.config.js` 文件名拼写错误（应为 `tailwindcss.config.js`），且使用了 Tailwind v3 的 `content` / `theme.extend.colors` 配置语法，而项目安装的是 Tailwind CSS v4（`^4.1.17`），v4 使用纯 CSS 配置方式（`@theme` 指令）。**自定义颜色（primary、background-light 等）大概率没有生效。**

**建议**：
- 删除 `taildwindcss.config.js`，将自定义 token 迁移到 `src/style.css` 中使用 `@theme` 指令：
  ```css
  @theme {
    --color-primary: #2b7cee;
    --color-background-light: #f6f7f8;
    --color-background-dark: #101822;
  }
  ```
- 或者降级到 Tailwind CSS v3（如果团队更熟悉 v3 语法）

---

### 1.2 HTTP 请求层有三个函数，行为不一致且存在不可达代码

**文件**：`src/utils/http.js`

| 函数 | 401 处理 | 错误返回方式 | 
|------|----------|-------------|
| `requestWithoutAuth` | 有 401 判断 | 返回错误对象 |
| `fetchWithAuth` | **不可达**（`!res.ok` 已提前 return） | throw 抛出异常 |
| `request` | 通过 `handleHttpError` | 返回 `{ success, message }` |

- `fetchWithAuth` 第 127 行的 `if (res.status === 401)` 永远无法执行（因为第 121 行 `!res.ok` 已经拦截了所有非 2xx 状态码）
- `media.js` 中 `saveMedia` / `deleteMedia` **绕过了统一的 HTTP 工具函数**，直接用原生 `fetch` 手写了整个请求逻辑（超时、错误处理），完全重复了 `http.js` 的功能
- 调用方无法预知 API 函数到底返回什么格式：有的返回原始 response，有的返回 `{success, data}`，有的直接 throw

**建议**：
- 统一为一个请求函数，通过参数控制认证行为
- 修复 `fetchWithAuth` 中不可达的 401 判断
- 将 `media.js` 中的 `saveMedia` / `deleteMedia` 改为复用 `http.js` 的 `fetchWithAuth`
- 推荐的做法：所有 API 函数统一返回 `{code, data, message}`，调用方统一判断 `res.code === 0`

---

### 1.3 NavigationBar.vue 存在未导入的变量引用

**文件**：`src/layout/components/NavigationBar.vue`

```js
// 第 24 行：import { Expand, Fold } from "@element-plus/icons-vue" — 但模板中未使用
// 第 50 行：ElMessage.success("退出成功") — ElMessage 未导入！
// 第 56 行：sessionStorage.clear() — 与 Pinia persistedstate 的 localStorage 不一致
```

- 退出登录逻辑应仅调用 `clearUser()` (Pinia persisted 会自动清除 localStorage)，删除手动操作 `sessionStorage`/`localStorage` 的代码

## 二、架构层面优化

### 2.1 不支持 TypeScript

项目 100% 使用 JavaScript，没有任何 `.ts` 文件。对于 82 个组件的系统级项目，缺乏类型安全会导致：
- 组件 props 没有类型约束（大量使用 `defineProps` 无类型标注）
- API 返回值没有类型，调用方需要猜测数据结构
- 重构时容易引入运行时错误

**建议**：渐进式迁移为 TypeScript（Vite + Vue 3 原生支持），优先从 `stores`、`apis`、`utils` 开始。

---

### 2.2 路由系统复杂度高，逻辑重复

**文件**：`src/router/index.js`（190 行）+ `src/layout/components/SideBar.vue`（355 行）

- 角色权限判断逻辑在**路由守卫**（`beforeEach`）和**侧栏渲染**（`accessibleRoutes` computed）中各写了一遍
- SideBar 的 `accessibleRoutes` 计算属性极其复杂（135 行），涉及角色过滤、路由去重、子路由处理、页面列表动态注入
- `utils/index.js` 中的 `addProtectedRoutes` / `resetRoutes` 是空函数（仅为保留向后兼容），说明经历过重构但留下了死代码

**建议**：
- 将角色判断逻辑抽取到独立的 composable（如 `usePermission.js`），路由守卫和侧栏共用
- SideBar 的菜单渲染逻辑拆分为独立组件
- 删除 `addProtectedRoutes` / `resetRoutes` 空函数及相关调用

---

### 2.3 全局 Store 存在冗余状态

**文件**：`src/stores/global.js`

```js
const user = ref(null);
const access_token = ref("");
const isLogin = ref(false);  // ← 冗余：可通过 !!access_token 推导
```

- `isLogin` 与 `access_token` 存在状态冗余，容易产生不一致
- `persist: true` 会将整个 store（包括 `access_token`）以明文存入 localStorage，存在安全风险

**建议**：
- 删除 `isLogin`，改为 `computed(() => !!access_token.value)`
- 考虑仅持久化必要的字段，或对敏感字段加密存储

---

### 2.4 大量 Field 组件缺乏抽象，重复代码多

**文件**：`src/components/Field/` 下共 50+ 个组件

- 每个组件都独立实现了图片上传、富文本编辑、表单交互等逻辑
- 若需要全局修改某个交互行为（如上传错误提示），需修改 50+ 个文件

**建议**：
- 抽取通用组合式函数（composables），如 `useImageField`、`useTextField`、`useMediaLibrary`
- 对于结构相似的组件（如所有 icon-box 变体），考虑合并为一个通用组件 + 配置驱动

---

## 三、性能优化

### 3.1 媒体库图片解析存在性能瓶颈

**文件**：`src/utils/mediaResolve.js`

```js
// 第 29 行：最坏情况下循环 8 页 × 100 条 = 800 次网络请求
for (let page = 1; page <= 8; page++) {
  res = await getMediaList({ site_id: siteId, page, page_size: 100 });
  // ...
}
```

**建议**：后端提供按 attachment_id 精确查询接口；如果前端必须查找，至少加上缓存，避免同一 attachment_id 重复遍历。

---

### 3.2 大列表未使用虚拟滚动

**文件**：`src/views/admin/List.vue`、`src/views/admin/UserList.vue`（720 行）

- 站点列表和用户列表直接使用 `el-table` 渲染全量数据
- 用户列表已有分页，但前端过滤 `filteredTableData`（当前直接返回 tableData）仍是一次性渲染

**建议**：
- 确认分页是否在后端进行（`getSiteList` 支持分页参数），避免前端一次性加载全量数据
- 如果需要渲染超过 100 条数据，考虑使用 Element Plus 的 `el-table-v2`（虚拟化表格）或 `vue-virtual-scroller`

---

### 3.3 所有路由组件无条件懒加载

**文件**：`src/router/index.js`

```js
component: () => import("@/views/admin/List.vue"),  // 已使用懒加载 ✓
```

虽然已使用动态 import，但没有针对性的预加载策略（如高频页面使用 `/* webpackPrefetch: true */`）。

**建议**：对用户登录后最可能访问的首页（如 `/admin/list` 或 `/siteInfo`）添加 prefetch 注释。

---

### 3.4 Quill 编辑器全局引入 CSS

**文件**：`src/main.js` + `src/components/QuillEditor.vue`

```js
// main.js：全局引入
import "quill/dist/quill.core.css";

// QuillEditor.vue：又引入了一次
import "quill/dist/quill.snow.css";
```

`quill.core.css` 和 `quill.snow.css` 都全局引入，增加了初始加载体积。

**建议**：仅在 `QuillEditor.vue` 中按需引入，避免全局 CSS。

## 五、安全性

### 5.1 access_token 明文存储在 localStorage

**文件**：`src/stores/global.js`

Pinia persistedstate 默认将整个 store 序列化为 JSON 存入 localStorage，包括 `access_token`。任何 XSS 攻击都可能导致 token 泄露。

**建议**：
- 使用 `pinia-plugin-persistedstate` 的 `paths` 配置仅持久化非敏感字段
- 或将 token 存储在 httpOnly cookie 中（需要后端配合）
- 短期方案：对 token 做简单的对称加密后再存储

---

**文件**：`.gitignore`

```gitignore
dist          # 第1行
node_modules  # 第2行
...
node_modules  # 第14行（重复）
dist          # 第15行（重复）
```

**建议**：清理重复项。

## 六、开发体验

### 6.1 缺少环境变量管理

项目中没有 `.env` / `.env.development` / `.env.production` 文件，API 地址硬编码在 `config/index.js` 中。

**建议**：
```bash
# .env.development
VITE_API_BASE_URL=https://apitest.yhct.site

# .env.production
VITE_API_BASE_URL=/fe
```

然后在 `config/index.js` 中使用 `import.meta.env.VITE_API_BASE_URL`。

---

### 6.2 缺少测试

项目没有任何测试文件（单元测试 / 组件测试 / E2E 测试）。

**建议**：至少为核心逻辑添加单元测试：
- `utils/dataExtractor.js` — 数据提取/同步逻辑
- `utils/http.js` — 请求封装
- 登录流程 E2E 测试

---

系统面向中文用户，应修改为 `lang="zh-CN"`。
